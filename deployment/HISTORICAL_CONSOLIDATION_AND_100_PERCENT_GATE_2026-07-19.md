## 5. Historical conflicts and corrections

### Conflict A: 100 percent completion certificates

Several historical documents claim that hundreds of modules were complete, all tests passed, zero gaps remained and the system was fully deployed.

Current evidence contradicts treating those claims as present truth because:

- accessible historical code includes explicit placeholders and pass-through methods;
- some sync integrations report zero work or contain comments saying implementation remains future work;
- current production endpoint reachability is unverified;
- Gmail OAuth is unresolved;
- the runtime worker remains a placeholder;
- source discovery is implemented only as an isolated branch module;
- CI execution evidence has not yet been returned;
- current source denominator and raw-export reconciliation are incomplete.

Decision: preserve these reports as `HISTORICAL_ASSERTION_SUPERSEDED_BY_CURRENT_EVIDENCE`.

### Conflict B: all conversations and messages migrated

Historical reports assert that all conversations and messages were migrated across multiple platforms. Available code proves parser/importer intent and partial implementation, but does not prove that every source export was present, parsed, reconciled and loaded successfully.

Decision: classify as `IMPLEMENTATION_CLAIM_WITHOUT_COMPLETE_RUN_EVIDENCE` until source counts, hashes, parse logs and target-database counts reconcile.

### Conflict C: full live intelligence

Historical trackers claim live data, continuous learning and automatic update loops. Some code excerpts contain placeholder REST handlers, placeholder sync methods, fixed confidence values and comments stating actual integration would be added later.

Decision: classify individual modules from repository/runtime evidence, not headings or certificates.
